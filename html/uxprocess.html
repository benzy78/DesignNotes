受託案件の場合に考慮すべきUX設計プロセス

クライアント企業の事業戦略と背景・目的の理解

  • なぜこのプロジェクトが立ち上がったのか
  • 事業上のKPI、経営課題、競合との戦いとプロジェクトの裏にある事情
  • 現場の担当者と経営層の温度差も要チェック(トップダウンか現場規定かなど)

ステークホルダー構造の把握

  • 社内の誰が意思決定者か？ (決裁権限の所在を明らかに)
  • 現場担当者と経営層で意見が違うことも多いため、誰のUXなのかを意識
  • 承認プロセスも事前に知っておくと提案の仕方が変わる

ユーザーリサーチの許諾・制約

  • ユーザー設定がクライアント企業側にあることが多い
  • インタビュー調整、調査対象の制限、個人情報の扱いなどに事前準備が必要
  • 代替手段（社内ユーザー、代理、既存データ活用）も検討

成果物に対する期待値コントロール

  • 何をいつまでにどう納品するか明確に
  • クライアントのふわっとした依頼に対し、設計の意義を言語化、可視化して伝える必要
  • UIだけでなく、UXプロセスを含めるなら、ペルソナ、カスタマージャーニーユーザ、課題、仮説などを納品範囲に含めるか確認

課題認識・仮説立て/ヒアリング

目的

  • プロジェクトの出発点として、「何を解決すべきか」を明らかにするため
  • クライアントや事業側が持つビジネス課題と背景の整理
  • ユーザーの問題を最初の仮説レベルで定義し、後続の調査設計の方向性を決めるため
  • 現時点での関係者間の認識のズレをなくす

内容

  • クライアントや関係者へのヒアリング
    • 事業目的、ビジネスモデル、KPI
    • 想定しているユーザー像とその課題
    • 既存プロダクト/競合との違い
    • 技術的・組織的制約（期間、予算、体制など）
  • 仮説の立案
    • ユーザーは〜という状況でXXという課題を感じているのでは？
    • 現状のプロダクトは〜だから使いづらく、XXな改善が必要では？
  • 問いの明確化（探索型調査のガイドとなる問い）
    • なぜユーザーは〜という行動をとるのか
    • 現在の選択ではなぜ満足していないのか

ターゲット層の暫定的な設定

目的

  • ユーザーリサーチの対象を絞るための仮設定
  • 無差別にリサーチするのではなく、想定すべき主要ユーザー像に焦点を当てる
  • 以後のペルソナ設計やカスタマージャーニーの設計のベースとなる

内容

  • 関係者ヒアリングや既存データをもとに
    • 年齢層、職業、利用目的、行動パターン、利用シーンなどを整理
    • ユーザーをセグメント分けし、重要そうな層を優先
    • リサーチ対象ユーザーの条件を定義（例：毎月1回以上〇〇サービスを使う20代後半の会社員など）
  • この時点では仮説に基づいた設定にし、後で修正前提。より精緻な人物像は、後の調査で深掘り・確定していく。

前提知識の強化

UX設計において「設計以前の前提知識」をどれだけ掘り下げるかは、設計の精度や視野を決定づける。 事前に知るべき前提知識は大きく以下の6カテゴリ。

なお、以降のUX設計プロセスと重複する部分があるが、前提知識の強化の段階では、業界理解・仮説形成を目的に広く浅くデスクリサーチを中心に行い、詳細な深掘りはその後のフェーズで行うものとする。 以降のプロセスで重複する項目もここで行う理由は、第一に、的外れなペルソナ設計やCJ設計を避けるため（事前知識がないとそれっぽいけどリアルじゃない設計になりがち。） 第二に、仮設ベースで調査設計やヒアリング設計ができる（たとえば、QRオーダーは厨房の確認ミスが多いらしいと知っていれば、ヒアリング時に調理側の確認プロセスも重点項目になる）。 第三に、業界固有のKPIや成功事例を把握できる（例：飲食店なら回転率と平均滞在時間が重要と知っていれば、ユーザー行動とKPIの関係性を設計に練りこめる）。

1. 業界知識・ビジネスモデル

業界特有の制約・収益構造・慣習を知らないと机上の空論になりやすい。調べる項目は以下。

  • 業界の構造（例：飲食店→ホール/厨房/管理、本部/フランチャイズの構造など）
  • 商流・収益構造（例：売上・原価・人件費・利益率など）
  • 業界特有の課題（例：人手不足、回転率、衛生管理、発注ロス）
  • 法規制や慣行（例：インボイス制度、食品衛生法 など）

2. 利用シーン・業務フロー

UIはコンテキストで成り立つため、どこで・いつ・誰が・どう使うかを把握する必要がある。調べる項目は以下。

  • 実際の業務フロー（例：QRコード注文→調理→配膳→会計の流れ）
  • 使用環境（例：片手・厨房の湿気・騒音・時間帯・ピークタイム）
  • デバイス特性（スマホ/タブレット/PC？）
  • 現場でのボトルネック（例：確認漏れ、フローの重複、動線の悪さ）

3. ユーザー属性・モチベーション

ペルソナやジョブを構築する際の土台とするために行う。

  • 利用者の属性（年齢、職種、ITリテラシーなど）
  • ユーザーが達成したいこと（ジョブ理論に基づく動機）
  • 課題感・ペインポイント・諦めていること

4. ビジネスゴールとKPI

UX設計を課題解決に留めず、成果創出に昇華させるため。

  • プロダクトが追っているKGI/KPI（注文回数？回転率？人件費削減？）
  • ユーザー価値と事業価値が交わるポイント
  • NPS・LTV・CVRなどの改善対象指標

5. システム制約

実装負荷や連携要件を把握せずに設計すると、実現できないUIになりがちなため。

  • 既存システムとの連携（POS連携・レジ締め処理など）
  • ネットワーク要件（店内Wi-Fiなど）
  • ハードウェアの制限（読み取り機・タブレット型番など）

補足：To CとTo Bの違い

観点 To C向け To B向け（SaaSなど）
重視点 感情・印象・わかりやすさ 効率性・整合性・正確性
設計起点 ユーザーの生活文脈 業務プロセス・役割分担
特有課題 離脱率・直感的操作 アクセス権管理・フローの複雑さ
調査軸 ユーザー体験/行動心理 業務要件/組織構造/導入障壁

ユーザーリサーチ（探索型の定性調査）

設計初期のプロセスにおけるユーザリサーチは、探索型の定性調査が有効。

ユーザーリサーチの目的

設計プロセス初期に探索型の訂正調査を行う目的はいくつかある。 第一に、ユーザの本質的な課題や欲求を発見するため。 これは表面的な要望ではなく、潜在的なニーズや生活、業務文脈を掘り起こす。 第二に、 仮説の立脚点を得るため。ペルソナ、カスタマジャーニー、情報設計などの後の工程のUX設計の根拠を得る。 第三に、 ユーザ視点をチーム全体で共有するため。開発やビジネス型と認識を合わせるための一次情報として活用する。

探索型定性調査の手法

  1. ユーザーインタビュー：1on1で深掘り。自由回答でユーザーの言葉を引き出す。新しい課題の発見や感情の深掘りに適している。
  2. エスノグラフィー(行動観察)： 実際の行動を現場で観察する手法で、行動と発言のギャップ確認や現場特有の慣習を把握する際に適している。
  3. シャドーイング： ユーザに密着し、業務や操作をリアルタイムで記録する手法。店舗やオフィスなど業務系プロダクトに適している。
  4. ユーザドキュメント分析： ユーザが使っている業務マニュアル、メモ、ホワイトボードなどを収集する手。 アナログな業務フローや工夫の把握をするのに適している。
  5. デプスインタビュー(深層面接)： ユーザの背景、価値観まで掘り下げる手法。製品へのこだわり、心理的障壁の解明に適している。
  6. コンテキストインタビュー： ユーザの作業環境やタイミングでインタビューする手法。実行中の思考プロセスや判断基準の把握に適している。

実施時のポイント

  1. 仮説を持つが押し付けない： 誘導尋問にならないよう、仮説はあるが、柔軟に捨てる姿勢で臨むことが重要
  2. なぜ？を繰り返す： 表層の回答の奥にある行動権利を掘り下げることが重要。 ヒアリングが浅くならないように注意する
  3. 行動、感情、判断の3軸で聞く： 例えば、なぜそうしたか、その時どう思ったか、どんな選択肢と迷ったかといったことを深掘りする
  4. 中間成果物を即座に可視化共有する： インサイトメモ、インタビューボード、FigJamなどでチーム共有する
  5. 情報を鵜呑みにして設計しない： ユーザの発言は課題のヒントであり、解決案ではないと理解する

競合リサーチ

目的

  • 市場における立ち位置の把握：類似サービスがどのようなUI/UXを提供しているかを把握し、自プロダクトの差別化要素を検討する
  • ベストプラクティスの抽出：他社が既に解決しているUI/UX上の課題を学び、設計の参考にする
  • ユーザーの期待値を知る：競合サービスが標準的なUIとなっている場合、それがユーザーにとっての「当たり前」になる

調査対象の選定

  • 直接競合：同じニーズやターゲットを持つサービス（例：飲食店向け注文アプリ）
  • 間接競合：同じ目的を別アプローチで達成するサービス（例：LINE注文、電話予約など）
  • 参考プロダクト：業界は違っても、UI/UXのアプローチが優れているサービス（例：SlackやNotionなど）

主な調査手法

  • 実際に利用し、操作フロー、フィードバック、UIコンポーネントの挙動などを体験
  • UIキャプチャ/画面設計の分析：画面の構成、導線、CTA配置、配色などを観察・比較
  • レビュー分析：App StoreやGoogleレビューなどからUX課題・満足点を抽出
  • ユーザー視点でのタスクベース評価：「◯◯したいときに、どのように導かれているか」など、特定のユースケースで体験を比較

分析観点

  • 情報設計：ナビゲーションの構造、階層の深さ、情報の並べ方
  • インタラクション設計：フォーム、入力補助、トグル、フィードバックの設計
  • ビジュアルデザイン：配色、アイコンの使い方、フォント、視線誘導の工夫
  • フロー設計：ユーザーが目的を達成するまでのステップの長さと分かりやすさ
  • エラー処理：入力ミスや通信エラーに対する案内や回復支援のUI
  • 利用開始のしやすさ：オンボーディング、チュートリアル、初期設定の誘導

以上の分析から、競合の弱み、強み、差別化ポイントを明確化する。

ペルソナ設計

目的

  • ユーザー増の具体化：抽象的な「ターゲット層」ではなく、具体的な人物像（仮想ユーザー）を定義することで、関係者間で共通の理解を持つ
  • UX設計の意思決定の拠り所：「このペルソナならこの導線のほうが使いやすいのでは？」など、設計の際の判断軸や優先順位づけに使う
  • ユーザーの課題・動機・背景の深掘り：行動や思考の背景にある動機、制約、習慣、価値観を明らかにし、UX改善のヒントを得る
  • 感情的共感の促進：開発・ビジネスサイドの関係者がユーザー視点に立ちやすくするための手段でもある

内容

ペルソナの構成要素は以下。 必ずしも全て埋める必要はなく、プロジェクトのフェーズやリソースに応じて柔軟に構成する。

  • 名前・年齢・性別
  • 職業・生活環境
  • デジタルリテラシー
  • 行動パターン
  • 主な課題（Pain）
  • 期待している価値（Gain）
  • 目的・動機・価値観
  • 使用シーン
  • 発言・思考の仮想セリフ

情報の収集源

  • ユーザーインタビュー
  • アンケート結果
  • 既存顧客データ、アクセス解析
  • カスタマーサポートや営業担当のヒアリング

注意点

  • ステレオタイプ化に注意：ペルソナは「理想ユーザー」ではなく、「実際にいそうな典型的なユーザー像」をベースに構築
  • 一人ではなく複数設計することが多い：主要ユースケースやニーズが大きく異なる場合は複数ペルソナを設け、優先ペルソナ（主対象）を決める
  • 設計のためのツールであり、絶対的な人物像ではない。リサーチに基づいて、更新される前提で設計する。

カスタマージャーニーの詳細

カスタマージャーニーとは、ユーザーが目的を達成するまでの体験の流れを可視化し、感情の起伏や課題ポイントを把握するための手法。

以下がQRコードオーダアプリのCJ例

フェーズ 行動 感情 タッチポイント 課題・機会
来店・着席 席に座る 期待とやや不安 QRコード案内表示 QRコードの案内がわかりづらいと不安に
QR読み取り スマホで読み取る 少し戸惑う QRコードリンク 読み取りミス、アプリのダウンロード要求があると離脱リスクに
メニュー閲覧 カテゴリからメニューを探す ワクワクするが、探しにくいとストレス メニュー画面 カテゴリ分類、検索性、一覧性の良さが重要
注文確定 商品を選び注文を送信 安心・達成感 注文カート・注文確定ボタン 注文ミスを防ぐ確認やインタラクションが必要
食事中 待つ/食べる 満足感 注文完了後の待機 状況がわからないと不安。注文がちゃんとされたのか確認できるような画面が必要
支払い・退店 会計して帰る スムーズな満足感 会計フロー モバイル決済誘導があるとさらに便利

課題整理と課題解決策の設計

「課題整理」は、リサーチをもとに抽出された情報を構造化し、真に解決すべきコア課題を明確化する重要なフェーズ。

目的

  • 表層的な「声」や「事象」から、本質的な課題（Core Issue）を見抜く
  • UX改善のために、課題を体系的・論理的に整理し、設計の方向性を決定する
  • 検討すべき課題の優先順位をつけるためのベースをつくる

課題整理の手順

  1. 情報の収集・抽出
    • ユーザーの発言内容・行動・感情などを網羅的にリストアップ
    • 「○○がわかりにくい」「△△の入力に手間取った」など、具体的な困りごとや不満点を抽出
  2. 解釈・構造化（インサイトの抽出）
    • 抽出された声を意味ごとにグルーピング（アフィニティダイアグラムなどを用いる）
    • 背景にある本質的な課題（Why）を深掘りして明文化（例：「情報が見つからない」→「情報設計がユーザー視点になっていない」）
  3. 課題の構造化と分類
    • 抽象度に応じて階層化（例：大課題→中課題→具体的な改善点）
    • 場合によっては、ユーザーの課題 / ビジネス上の課題 / 技術的制約など、視点ごとに分類
  4. 課題の優先度づけ
    • 影響度×実現可能性のマトリクスを使って優先順位を可視化
    • または、KJ法/4象限分析（重要×緊急）/CJ上での出現頻度などを用いる
  5. 各課題に対して、解決策を検討する

要件定義

ユーザーリサーチや課題整理・解決案の検討を経て、プロダクトが実装すべき機能・仕様・制約条件を言語化・構造化するプロセス。

要件の分類

  • 機能要件：実装すべき機能やUI挙動の定義（例：お気に入り登録機能、注文履歴の閲覧など）
  • 非機能要件：性能・セキュリティ・可用性・アクセシビリティなど、体験を支える仕様（例：3秒以内に表示、WCAG AA準拠）
  • 制約条件：技術・予算・期間・社内体制などの外部条件（例：iOSのみで先行リリース、社内API利用必須）

要件定義の手順

  1. プロダクトの前提とスコープの確認
  2. 大まかな画面構成とユーザーフローを整理
  3. 上記をもとに、要件リストの構築（機能・非機能・制約）：具体的な要件を記述・分類・優先順位づけ。OOUIの手法で整理するのも良い。
  4. 要件の見直し・スコープ整理：設計・開発・テスト工数のバランスをとりながら、どこまでを初期リリースで対応するかを決定

＊要件定義・情報設計・ワイヤフレーム作成の手順はこだわりすぎなくて良い。 目に見える形にして初めて気づくことも多いので、順番が前後しても良いし、各ステップを行ったり来たりして良い。

情報設計

ユーザーが必要な情報に迷わずたどり着けるように、情報や機能の構造・ラベル・導線を整理・設計する工程。

情報設計の手順

  1. コンテンツ・機能の一覧作成
  2. 情報の分類・構造化
  3. サイトマップ・画面フローの作成
  4. ラベリング：ボタン名などの文言をユーザー視点でわかりやすく設計
  5. ナビゲーション設計（導線とUI構造）：メニュー構成、パンくず、タブ構成など

ワイヤフレーム/プロトタイプ作成

UX設計におけるワイヤーフレーム作成は、情報設計や要件定義をもとに、各画面のレイアウトや要素配置、導線、構造を視覚化する工程。

作成手順

  1. 前提整理：要件定義、情報設計等
  2. 画面単位で必要な要素を洗い出す
  3. ラフレイアウトを作成
  4. ユーザーフロー・導線をつなぐ（プロトタイプ）：各ページ間の遷移をつなぐ

*OOUIは設計初期の段階で使うべきか？ UX設計初期では、「このプロダクトで何を解決するか」「どんなフローになるか」が主で、 OOUIの「対象（名詞）中心」の思考はまだ馴染まないので、ある程度の画面遷移・フロー・機能要件が仮置きできた段階で、 タスク指向UIになっている部分をOOUI的にリファクタリングする方が現実的。

検証・フィードバック・再設計

単なる「デザインの見た目チェック」ではなく、ユーザーが本当に課題を解決できるか／価値を感じるかを確認するフェーズ。

検証すべきこと

  1. タスク達成性：ユーザーが意図した操作を迷わず達成できるか
  2. 情報構造の妥当性：場所、順番、分類は直感的か
  3. 導線・フローの自然さ：遷移や操作がストレスにならないか。特にタスク指向UIの部分をOOUIにするかの検討
  4. 提供価値の実感：コア体験が「便利・快適・有益」と感じられるか
  5. 不安・誤解の有無：表現や配置に曖昧さや誤解がないか
  6. 使用意図とのギャップ：想定ユーザーの利用シーンとマッチしているか

上記の内容を検証し、必要があればワイヤフレームなどの再設計を行う。

なお、設計プロセスとしては、コア機能に絞って調査や設計を行い、MVPを作成して徐々にスコープを広げていくという手法をとることもある。 プロジェクトの性質（0→1か拡張期なのか）によって適切な方法を選択する。
